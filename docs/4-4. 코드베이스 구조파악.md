# 4-4. 코드베이스 구조 파악

`legacy-portal` 프로젝트의 패키지 구조와 각 컴포넌트의 책임을 정리한다.
(작성 시점 코드 기준 스냅샷 — 리팩토링이 진행되면 세부 내용은 달라질 수 있다.)

## 1. 개요

- KT DS 「클린코드 & 리팩토링 실전」 과정의 실습용 사내 업무 포털
- 도메인: **결재(approval)** · **공지(notice)** · **일정(schedule)** · **사용자(user)**
- 공통 모듈: **common** (메일 발송, 감사 로그)
- Java 21 · Spring Boot 3.2 (starter-web, data-jpa) · JUnit 5 · Maven Wrapper
- DB: `DataSourceConfig`가 기동 시 MariaDB(localhost:3306/portal) 접속을 2초 타임아웃으로 시도해보고,
  실패하면 자동으로 H2 인메모리(`MODE=LEGACY`)로 폴백 — 별도 프로파일/URL 설정 불필요
- `spring.jpa.hibernate.ddl-auto=create-drop` → 기동할 때마다 스키마 재생성, 데이터는 `DataSeeder`가 사용자 4명만 시딩

## 2. 패키지 구조

```
com.ktds.portal
├── PortalApplication      # @SpringBootApplication 진입점
├── DataSourceConfig       # DB 자동 선택(MariaDB 우선, 실패 시 H2)
├── DataSeeder             # CommandLineRunner - 초기 사용자 4명 시딩
├── approval/              # 결재 도메인
│   ├── Approval            (Entity)
│   ├── ApprovalRepository  (JpaRepository)
│   ├── ApprovalService     (핵심 로직 - 이 프로젝트의 "안티패턴 교재")
│   └── ApprovalController  (REST, /api/approvals)
├── notice/                 # 공지 도메인
│   ├── Notice               (Entity)
│   ├── NoticeRepository     (JpaRepository)
│   └── NoticeService        (Controller 없음 — 아직 REST로 노출 안 됨)
├── schedule/               # 일정 도메인
│   ├── Schedule              (Entity)
│   ├── ScheduleRepository    (JpaRepository)
│   └── ScheduleService       (Controller 없음 — 아직 REST로 노출 안 됨)
├── user/                   # 사용자 도메인
│   ├── User                 (Entity)
│   ├── UserRepository       (JpaRepository)
│   └── UserController       (REST, /api/users — Service 없이 Repository 직접 사용)
└── common/                 # 공통 모듈 (도메인에 종속되지 않음)
    ├── SmtpMailSender        (메일 발송 - 실제로는 콘솔 출력)
    └── FileAuditLogger       (감사 로그 - 실제로는 콘솔 출력)
```

> **주의(비대칭 구조)**: 4개 도메인이 전부 같은 계층(Entity+Repository+Service(+Controller))을
> 갖는 게 아니다. `approval`만 Controller까지 완비되어 있고, `notice`/`schedule`은 Service까지만
> 있고 Controller가 없다(REST로 아직 노출되지 않음). `user`는 반대로 Service 없이 Controller가
> Repository를 직접 호출한다. 리팩토링/기능 추가 시 이 비대칭을 전제하고 접근해야 한다.

## 3. 도메인별 책임

### 3.1 approval (결재) — 이 프로젝트의 핵심 실습 대상

| 컴포넌트 | 책임 |
|---|---|
| `Approval` (Entity) | 결재 문서 데이터. `status`(0=임시저장·1=상신·2=승인·3=반려·9=취소), `type`(1=지출·2=휴가·3=구매·4=기타), `priority`(1=낮음·2=보통·3=높음)를 모두 `int`로 보유. 전부 public getter/setter만 있는 빈약한 도메인 모델. |
| `ApprovalRepository` | `findByDrafterId`(내 기안함) / `findByApproverId`(내 결재함) 조회만 제공하는 JpaRepository. |
| `ApprovalService` | **생성 + 상태전이(상신/승인/반려/취소) + 권한판정 + 메일발송 + 감사로그 + 상태라벨/금액등급 계산**을 전부 떠맡는 God Class. `processApproval()` 한 메서드가 action 코드(1/2/3/9)로 분기하는 100줄 이상의 중첩 if 블록. `SmtpMailSender`/`FileAuditLogger`를 생성자 주입 없이 직접 `new`. |
| `ApprovalController` | `/api/approvals` REST 엔드포인트. 요청 DTO 없이 `Map<String,Object>`로 파라미터를 그대로 받아 서비스로 전달(검증 없음, 오탈자여도 컴파일 통과). |

**상태 전이 규칙(레거시 동작, 결함 포함 그대로 보존 대상)**
- 상신(action=1): `상태==임시저장(0)`일 때만 → 상신(1). 지출(type=1)이면서 금액 100만원 이상이면 우선순위를 자동으로 "높음(3)"으로 상향.
- 승인(action=2): `상태==상신(1)` + 본인이 결재자 + `role>=2`(팀장 이상)일 때만 → 승인(2).
- 반려(action=3): 승인과 조건 동일 → 반려(3), 사유 기록.
- 취소(action=9): `상태==임시저장(0) 또는 상신(1)` + 본인이 기안자일 때만 → 취소(9).
- **조건 불충족 시 예외 없이 조용히 무시**(아무 것도 안 바뀜, 호출자는 실패 여부를 알 수 없음) — 나쁜 설계지만 리팩토링 단계에서는 그대로 보존해야 하는 관찰된 동작.

### 3.2 notice (공지)

| 컴포넌트 | 책임 |
|---|---|
| `Notice` (Entity) | 공지 데이터. `status`(0=임시·1=게시·9=내림), `category`(1=일반·2=긴급·3=인사)를 `int`로 보유. |
| `NoticeRepository` | 커스텀 조회 메서드 없는 순수 JpaRepository. |
| `NoticeService` | 생성 + 게시(publish) + 상태라벨 계산. `publish()`는 `role>=2`(팀장 이상)일 때만 임시(0)→게시(1) 전환, 긴급(category=2) 공지면 전 직원에게 메일 발송. `ApprovalService`와 거의 동일한 감사 로그/메일 본문 생성 로직이 중복돼 있음(공통 모듈 추출 실습 대상). |
| Controller 없음 | 현재 REST API로 노출되지 않음. |

### 3.3 schedule (일정)

| 컴포넌트 | 책임 |
|---|---|
| `Schedule` (Entity) | 일정 데이터. `status`(0=예정·1=확정·9=취소)를 `int`로 보유. |
| `ScheduleRepository` | `findByOwnerId`(소유자별 일정 조회) 제공. |
| `ScheduleService` | 생성(제목/시간 검증 + 동일 소유자 시간 겹침 검사 + 저장 + 감사로그) + 확정(confirm, 본인 소유 + 예정 상태일 때만). 겹침 검사는 boolean 플래그(`flag1`)를 두고 전체 일정을 순회하며 판정 — 가독성이 낮아 메서드 추출 대상. 검증 실패 시 `null` 반환(호출자가 실패 원인을 알 수 없음). |
| Controller 없음 | 현재 REST API로 노출되지 않음. |

### 3.4 user (사용자)

| 컴포넌트 | 책임 |
|---|---|
| `User` (Entity) | 사용자 정보(이름/이메일/부서) + `role`(1=사원·2=팀장·3=임원, `int`). 이 `role` 값이 approval/notice 서비스의 권한 판정(`role>=2`)에 매직넘버로 직접 쓰인다. |
| `UserRepository` | 커스텀 메서드 없는 순수 JpaRepository. |
| `UserController` | `/api/users` — 전체 사용자 목록 조회만 제공(데모 프론트엔드의 로그인/사용자 선택 화면용). Service 계층 없이 Repository를 직접 호출. |

### 3.5 common (공통 모듈)

| 컴포넌트 | 책임 |
|---|---|
| `SmtpMailSender` | 메일 발송. 실제 SMTP 연동 없이 콘솔에 출력만 함(실습용 스텁). 인터페이스가 없어 테스트 시 교체 불가. |
| `FileAuditLogger` | 감사 로그 기록. 실제 파일 저장 없이 콘솔에 출력만 함(실습용 스텁). 마찬가지로 구체 클래스 직접 의존. |

두 클래스 모두 `approval`/`notice`/`schedule` 서비스에서 생성자 주입 없이 `new`로 직접 생성해 사용 중 → DIP 위반, 리팩토링 대상.

## 4. 서비스 간 공통 패턴 (리팩토링 관찰 포인트)

- **상태 필드가 전부 매직넘버 `int`** (`Approval.status/type/priority`, `Notice.status/category`, `Schedule.status`, `User.role`) — enum화 후보. 단 DB 저장값(정수)은 그대로 유지해야 함(`CLAUDE.md` 불변 규칙).
- **권한 판정 `role >= 2`가 `ApprovalService`·`NoticeService`에 그대로 중복**되어 있음.
- **감사 로그 기록 코드**(`now` 포맷팅 + `audit.write(...)`)가 `ApprovalService`/`NoticeService`/`ScheduleService` 세 곳에 거의 동일하게 복붙되어 있음 — 공통 모듈 추출 실습의 핵심 대상.
- **메일 본문 생성 패턴**(인사말 + 제목/사유 조합)이 `ApprovalService`(3곳)·`NoticeService`(1곳)에 반복.
- **조회 실패/검증 실패 시 예외 대신 `null` 반환 또는 조용히 무시**하는 패턴이 세 서비스 모두에 존재 — 관찰된 레거시 동작이므로 리팩토링 단계에서는 그대로 보존, 별도 재설계 단계에서 다룰 대상.
- **테스트 코드 없음** — 모든 도메인에 안전망 부재. 리팩토링 전 특성화 테스트 작성이 선행돼야 함.
