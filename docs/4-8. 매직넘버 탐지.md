# 4-8. 매직넘버 탐지

전체 소스에서 발견한 매직넘버(의미를 숫자로만 표현한 리터럴)를 도메인별로 정리하고,
주석·코드 사용 맥락으로부터 의미를 추정했다. (탐지만 수행 — enum화 등 실제 리팩토링은
별도 단계. `docs/4-4`, `docs/4-5`, `docs/4-7` 참고)

## 1. `Approval.status` (결재 상태)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 0 | 임시저장 (DRAFT) | `create()`가 생성 직후 `setStatus(0)` — 주석 "0=임시저장" |
| 1 | 상신 (SUBMITTED) | `processApproval` action=1 처리 후 `setStatus(1)` |
| 2 | 승인 (APPROVED) | action=2(승인) 처리 후 `setStatus(2)` |
| 3 | 반려 (REJECTED) | action=3(반려) 처리 후 `setStatus(3)`, `rejectReason` 동시 기록 |
| 9 | 취소 (CANCELED) | action=9(취소) 처리 후 `setStatus(9)` — 4~8은 정의되지 않은 결번 |

등장 위치: `Approval.java:26`(필드 주석), `ApprovalService.java:53,97,115,134,153,172-177`(`statusLabel`).

## 2. `Approval.type` (결재 종류)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1 | 지출 | `ApprovalService.processApproval()`의 `d.getType()==1` 분기 주석 "지출" + 금액 임계값 규칙과 결합 |
| 2 | 휴가 | 주석에서만 언급, 실제 분기 로직 없음 |
| 3 | 구매 | 주석에서만 언급, 실제 분기 로직 없음 |
| 4 | 기타 | 주석에서만 언급, 실제 분기 로직 없음 |

등장 위치: `Approval.java:25`, `ApprovalController.java:27`(요청 파라미터 주석), `ApprovalService.java:94`.
→ **관찰**: 정의된 4개 값 중 실제로 서비스 로직이 분기하는 값은 `1`(지출)뿐이다. 2/3/4는
저장은 되지만 코드상 의미가 반영된 곳이 없다(사각지대) — 값은 있지만 규칙이 없는 상태.

## 3. `Approval.priority` (우선순위)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1 | 낮음 | 주석 "1 낮음" |
| 2 | 보통 | 요청 시 기본값으로 사용 가능 |
| 3 | 높음 | `create()`에서 `urgent==true`일 때 강제 설정, `processApproval()`에서 지출 100만원 이상 시 자동 상향 |

등장 위치: `Approval.java:27`, `ApprovalController.java:28`, `ApprovalService.java:52,95`.
→ **관찰**: "3이 왜 높음인지"는 클래스 어디에도 정의되어 있지 않고 주석으로만 알 수 있다
(코드 스멜 문서의 지적과 동일).

## 4. `processApproval`의 `action` 파라미터

| 값 | 추정 의미 | `Approval.status`와의 관계 |
|---|---|---|
| 1 | 상신 요청 | 처리 결과 `status=1`과 **숫자가 동일** |
| 2 | 승인 요청 | 처리 결과 `status=2`와 **숫자가 동일** |
| 3 | 반려 요청 | 처리 결과 `status=3`과 **숫자가 동일** |
| 9 | 취소 요청 | 처리 결과 `status=9`와 **숫자가 동일** |

등장 위치: `ApprovalController.java:36-43`(요청 파라미터 주석), `ApprovalService.java:73,90,110,129,149`.

→ **흥미로운 관찰**: "요청 액션 코드"(action)와 "처리 결과 상태"(status)가 **개념적으로는
다른데(요청 vs 결과) 값의 숫자가 우연히 완전히 일치**한다. 이 때문에 코드만 보면 두 개념을
하나의 enum으로 합쳐도 될 것처럼 보이지만, "상신 요청(action)"과 "상신 상태(status)"는
읽는 관점이 다른 개념이다 — enum화할 때 이 둘을 무심코 하나의 타입으로 합치면 개념이
뒤섞일 수 있으니 주의가 필요한 지점으로 기록해둔다.

## 5. `User.role` (직급/권한)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1 | 사원 | 주석 "1=사원", `DataSeeder`가 김사원·최사원에 `1` 부여 |
| 2 | 팀장 | 주석 "2=팀장", `DataSeeder`가 박팀장에 `2` 부여 |
| 3 | 임원 | 주석 "3=임원", `DataSeeder`가 이임원에 `3` 부여 |

등장 위치: `User.java:19`, `DataSeeder.java:24-27`, `ApprovalService.java:114,133`, `NoticeService.java:56`.

## 6. 권한 판정 `role >= 2` (파생 매직넘버)

`role` enum 값 자체 말고도, **비교 연산에 등장하는 `2`라는 상수**가 또 다른 매직넘버다.
"팀장 이상만 승인/게시 가능"이라는 도메인 규칙이 코드 두 곳(`ApprovalService`
승인·반려 분기, `NoticeService.publish`)에 `role >= 2`로 복붙되어 있다
(`docs/4-5` §3 "권한 판정 중복"과 동일 지점).

## 7. `Notice.status` (공지 상태)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 0 | 임시 (게시 전) | `create()`가 `setStatus(0)`으로 시작 |
| 1 | 게시 (PUBLISHED) | `publish()`가 `status==0`일 때만 `1`로 전환 |
| 9 | 내림 | 주석 "9=내림" — 실제로 이 값을 `9`로 세팅하는 로직은 코드 어디에도 없음 |

등장 위치: `Notice.java:19`, `NoticeService.java:37,58,78-81`.
→ **관찰**: `9`(내림)는 필드 주석과 `statusLabel()` 번역표에는 존재하지만, 상태를 `9`로
바꾸는 메서드 자체가 없다 — "정의되었지만 도달 불가능한 값"이다.

## 8. `Notice.category` (공지 분류)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 1 | 일반 | 주석 "1=일반", 별도 분기 없음(디폴트 취급) |
| 2 | 긴급 | `publish()`에서 `category==2`일 때 전 직원 메일 발송 |
| 3 | 인사 | 주석 "3=인사" — 실제 분기 로직 없음 |

→ **관찰**: `Approval.type`과 마찬가지로, 정의된 값 중 실제 로직이 반응하는 값은 `2`
(긴급) 하나뿐이다. `3`(인사)은 저장 가능하지만 아무 동작도 유발하지 않는다.

## 9. `Schedule.status` (일정 상태)

| 값 | 추정 의미 | 근거 |
|---|---|---|
| 0 | 예정 | `create()`가 `setStatus(0)`으로 시작 |
| 1 | 확정 (CONFIRMED) | `confirm()`이 `status==0`일 때만 `1`로 전환 |
| 9 | 취소 | `create()`의 겹침 검사에서 `status!=9`(취소된 일정은 겹침 검사 제외)로만 참조 — `Notice.status=9`와 마찬가지로 **`9`로 세팅하는 로직이 없다** |

등장 위치: `Schedule.java:18`, `ScheduleService.java:42,58,73-74`.
→ 세 도메인(Approval/Notice/Schedule) 모두 `9`를 "취소/내림" 계열로 정의해두고 있지만,
실제로 `9`를 쓰는(set) 코드가 있는 곳은 `Approval`뿐이고 `Notice`/`Schedule`은 값만
정의되고 전이 로직이 빠져있다.

## 10. 금액 임계값

| 값 | 등장 위치 | 추정 의미 |
|---|---|---|
| 1,000,000 (100만원) | `ApprovalService.processApproval()` 94행 | 지출(type=1) 결재 상신 시, 이 금액 이상이면 `priority`를 3(높음)으로 자동 상향 |
| 10,000,000 (1000만원) | `ApprovalService.amountGrade()` 184행 | 등급 "S" 기준 |
| 1,000,000 (100만원) | `ApprovalService.amountGrade()` 185행 | 등급 "A" 기준 |
| 100,000 (10만원) | `ApprovalService.amountGrade()` 186행 | 등급 "B" 기준 |

→ **관찰**: "우선순위 자동상향 기준"(94행)과 "금액등급 A 기준"(185행)이 **똑같이
1,000,000**이라는 값을 쓴다. 같은 규칙을 재사용한 것인지, 우연히 값이 겹친 것인지
코드만으로는 판단할 수 없다 — enum/상수화 시 두 값을 하나의 상수로 묶어도 되는지
도메인 담당자에게 반드시 확인이 필요한 지점.

## 11. (부가) 인프라 설정값

업무 규칙은 아니지만 하드코딩된 상수 두 개가 `DataSourceConfig`에 있다.

| 값 | 위치 | 의미 |
|---|---|---|
| 2 | `DriverManager.setLoginTimeout(2)` | MariaDB 연결 확인 타임아웃(초) — 주석에 "2초 안에 연결"로 명시되어 있어 의미는 이미 드러남 |
| 3306 | `MARIADB_URL` 문자열 내 | MariaDB 기본 포트 — 업계 표준 포트라 매직넘버라기보다 설정값에 가까움 |

업무 도메인 매직넘버(1~10번)와 달리 이 둘은 리스크가 낮아 참고로만 남긴다.

## 종합 관찰

- **정의는 됐지만 로직이 없는 값**(사각지대)이 세 곳: `Approval.type`의 2/3/4,
  `Notice.category`의 3(인사), `Notice.status`·`Schedule.status`의 9. enum화할 때
  "값은 있는데 동작이 없는" 이 케이스들을 실수로 새로 구현해 넣지 않도록 주의가 필요하다
  (동작 변경 금지 원칙상, 지금 동작이 없다는 사실 자체를 보존해야 함).
- **서로 다른 개념이 같은 숫자를 공유하는 경우**가 두 곳: `action`(요청) vs `status`(결과)의
  1/2/3/9, 그리고 우선순위 상향 기준(100만원) vs 금액등급 A 기준(100만원). enum/상수
  추출 시 이 두 그룹을 성급히 하나로 합치면 서로 다른 도메인 규칙이 뒤섞일 위험이 있다.
- **`role >= 2`처럼 필드 값 자체가 아닌 "비교 상수"도 매직넘버**라는 점 — enum화 대상은
  `status`/`type`/`role` 같은 필드뿐 아니라, 그 필드를 비교하는 리터럴(`>= 2`, `== 1000000`
  등)도 포함해서 봐야 전체 그림이 잡힌다.
