# 4-5. 중복 코드 탐지

`ApprovalService` · `NoticeService` · `ScheduleService` 3개 서비스에서 발견한 중복 코드를
패턴별로 정리한다. (탐지만 수행 — 아직 리팩토링/추출은 하지 않음. `docs/4-4` 참고)

## 요약

| # | 중복 패턴 | 발생 횟수 | 관련 클래스 |
|---|---|---|---|
| 1 | 협력 객체 직접 `new` (필드 선언) | 5 | Approval, Notice, Schedule |
| 2 | 감사 로그 기록 (타임스탬프 포맷 + `audit.write`) | 6 | Approval, Notice, Schedule |
| 3 | 권한 판정 `role >= 2` | 3 | Approval(×2), Notice(×1) |
| 4 | 메일 본문 생성 + 발송 | 4 | Approval(×3), Notice(×1) |
| 5 | "조회 → null이면 조용히 리턴" 가드 | 6 (엔티티 3 + 사용자 3) | Approval, Notice, Schedule |
| 6 | 상태코드 → 한글 라벨 변환(`statusLabel`) | 2 | Approval, Notice |

---

## 1. 협력 객체 직접 `new` (필드 선언 중복)

세 서비스 모두 `SmtpMailSender`/`FileAuditLogger`를 생성자 주입 없이 필드에서 직접 생성한다.
**완전히 동일한 두 줄**이 클래스마다 반복.

| 위치 | 코드 |
|---|---|
| `ApprovalService.java:37-38` | `private final SmtpMailSender mail = new SmtpMailSender();`<br>`private final FileAuditLogger audit = new FileAuditLogger();` |
| `NoticeService.java:24-25` | 위와 완전히 동일 |
| `ScheduleService.java:20` | `private final FileAuditLogger audit = new FileAuditLogger();` (mail 필드는 없음) |

→ DIP 위반(테스트에서 교체 불가) + 동일 초기화 코드 3중 복붙.

---

## 2. 감사 로그 기록 — "타임스탬프 포맷 + audit.write"

`yyyy-MM-dd HH:mm:ss` 포맷의 현재 시각을 만들어 `[시각] 이벤트 id=.. by=..` 문자열로
`audit.write()`에 넘기는 동일한 3줄짜리 블록이 6곳에 반복된다.

| 위치 | 이벤트 |
|---|---|
| `ApprovalService.java:62-65` (`create`) | `APPROVAL CREATE` — 인라인 반복(아래 `writeAudit()`가 있음에도 사용 안 함 → **불완전한 중복 제거**) |
| `ApprovalService.java:163-166` (`writeAudit` private 헬퍼) | `submit/approve/reject/cancel` 4곳에서 호출(108/125/145/156행) — 그 자체가 아래 두 서비스와 같은 모양의 코드 |
| `NoticeService.java:44-45` (`create`) | `NOTICE CREATE` — 인라인 |
| `NoticeService.java:69-70` (`publish`) | `NOTICE PUBLISH` — 인라인 |
| `ScheduleService.java:62-63` (`create`) | `SCHEDULE CREATE` — 인라인 |
| `ScheduleService.java:76-77` (`confirm`) | `SCHEDULE CONFIRM` — 인라인 |

공통 형태:
```java
String now = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
audit.write("[" + now + "] <이벤트명> id=" + <id> + " by=" + <userId>);
```
→ `ApprovalService`는 자체적으로 `writeAudit()`를 추출했지만 `create()`는 여전히 인라인이라
**부분적으로만 중복 제거된 상태**. `Notice`/`Schedule`은 추출 자체가 안 되어 있음.

## 3. 권한 판정 `role >= 2` (매직넘버 중복)

"팀장 이상만 가능"이라는 동일한 판정이 서로 다른 두 서비스, 같은 서비스 내 두 메서드에
그대로 복붙되어 있다.

| 위치 | 컨텍스트 |
|---|---|
| `ApprovalService.java:114` (`processApproval`, 승인 분기) | `if (u.getRole() >= 2) {` |
| `ApprovalService.java:133` (`processApproval`, 반려 분기) | 승인 분기와 **완전히 동일** — 같은 클래스, 같은 메서드 내 중복 |
| `NoticeService.java:56` (`publish`) | `if (u.getRole() >= 2) {` |

→ `role`의 의미(1=사원·2=팀장·3=임원)와 "2 이상 = 승인/게시 권한"이라는 도메인 규칙이
세 곳에 흩어져 있어, 등급 체계가 바뀌면 세 곳을 모두 찾아 고쳐야 한다.

## 4. 메일 본문 생성 + 발송

"안녕하세요 {수신자}님,\n{알림 문장}\n제목: {제목}" 형태의 본문을 만들고
`mail.send(수신자이메일, "[라벨] " + 제목, body)`를 호출하는 패턴이 4곳에 반복된다.

| 위치 | 컨텍스트 | 라벨 |
|---|---|---|
| `ApprovalService.java:102-107` (상신 시 결재자에게) | `approver` 대상 | `[결재요청]` |
| `ApprovalService.java:119-124` (승인 시 기안자에게) | `drafter` 대상 | `[결재승인]` |
| `ApprovalService.java:138-144` (반려 시 기안자에게) | `drafter` 대상, 사유 포함 | `[결재반려]` |
| `NoticeService.java:63-66` (긴급 공지 게시 시 전직원에게, 반복문 내부) | `member` 대상(전체 사용자 루프) | `[긴급공지]` |

공통 형태:
```java
String body = "안녕하세요 " + <수신자>.getName() + "님,\n"
        + "<알림 문장>\n제목: " + <엔티티>.getTitle()
        + (부가 정보 있으면 추가);
mail.send(<수신자>.getEmail(), "[라벨] " + <엔티티>.getTitle(), body);
```
→ 인사말/제목 라인은 완전히 동일한 문자열 조립 방식이고, 달라지는 부분은 "알림 문장"과
"라벨", 부가 정보(기안자ID·사유) 뿐이다.

## 5. "조회 → null이면 조용히 리턴" 가드 패턴

대상 엔티티와 사용자를 `findById(...).orElse(null)`로 조회한 뒤 null이면 예외 없이
그냥 `return`하는 가드 코드가 3개 서비스의 상태 변경 메서드마다 반복된다.

| 위치 | 엔티티 조회 가드 | 사용자 조회 가드 |
|---|---|---|
| `ApprovalService.processApproval` (76-84행) | `Approval d = repo.findById(id).orElse(null); if (d == null) return;` | `User u = userRepo.findById(userId).orElse(null); if (u == null) return;` |
| `NoticeService.publish` (50-53행) | `Notice n = repo.findById(id).orElse(null); if (n == null) return;` | 위와 동일한 모양 |
| `ScheduleService.confirm` (68-71행) | `Schedule sc = repo.findById(id).orElse(null); if (sc == null) return;` | 위와 동일한 모양 |

→ 특히 **사용자 조회 가드 두 줄은 세 클래스에서 글자 하나 다르지 않게 동일**하다.
→ `CLAUDE.md` 기준: "조용히 무시"는 레거시의 관찰된 동작이므로 리팩토링 단계에서 예외로
바꾸면 안 되고(동작 변경 금지), 대신 **가드 코드 자체의 구조적 중복만** 추출 대상으로 삼는다.

## 6. 상태코드 → 한글 라벨 변환 (`statusLabel`)

`status`(int)를 화면 표시용 한글 문자열로 바꾸는 if-else 체인이 `Approval`/`Notice`
양쪽에 거의 같은 모양으로 존재한다(값과 라벨 텍스트만 다름). `Schedule`에는 아직 이
메서드가 없다(비대칭 — 향후 추가 시 또 같은 모양이 복붙될 가능성이 높은 지점).

| 위치 | 매핑 |
|---|---|
| `ApprovalService.statusLabel` (169-179행) | 0→임시저장, 1→상신, 2→승인, 3→반려, 9→취소, else→알수없음 |
| `NoticeService.statusLabel` (76-82행) | 0→임시, 1→게시, 9→내림, else→알수없음 |

→ 값 자체는 도메인마다 다른 의미(예: 둘 다 `9`지만 하나는 "취소", 하나는 "내림")이므로
값 자체를 공유할 수는 없지만, "if-else 체인으로 상태→라벨 매핑" 이라는 **구조(shape)의
중복**은 각 도메인의 상태 enum이 스스로 라벨을 갖게 하는 방식(Rich Domain)으로 없앨 수 있다.

## 참고 — 이 문서의 용도

이 목록은 `CLAUDE.md`의 리팩토링 원칙에 따라 **탐지 결과를 먼저 고정**해두기 위한 것이다.
실제 추출(공통 모듈화, enum화 등)은 별도 단계에서 진행하며, 그 전에 여기 열거된 각 동작을
특성화 테스트로 먼저 고정한 뒤 리팩토링해야 한다(리팩토링 전후 테스트가 변경 없이 green).
