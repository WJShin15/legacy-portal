# 특성화 테스트 결과 — processApproval 커버리지

`ApprovalServiceProcessApprovalCharacterizationTest`(6개 테스트)가 `ApprovalService.processApproval`을
실제로 얼마나 exercise하는지 JaCoCo로 측정했다. (측정은 `org.jacoco:jacoco-maven-plugin:0.8.15`를
Maven 좌표로 1회성 실행한 것 — `pom.xml`에는 아무 것도 추가하지 않았다.)

## 커버리지 수치 (processApproval 메서드 단위)

| 지표 | 커버 | 전체 | 커버리지 |
|---|---|---|---|
| Line | 56 | 58 | 96.6% |
| Instruction | 246 | 253 | 97.2% |
| **Branch** | **30** | **48** | **62.5%** |
| Complexity(경로) | 8 | 25 | 32.0% |

**라인 커버리지(96.6%)는 높지만 분기 커버리지는 62.5%에 그친다.** 6개 테스트가 메서드의
거의 모든 "줄"은 한 번씩 지나가지만, if 조건의 참/거짓 양쪽을 다 짚지는 못했다는 뜻이다.
God Class·Long Method(`docs/4-7`)의 특성상 한 메서드 안에 조건 분기가 많아서 줄 커버리지만
보면 착시가 생기는 대표적인 사례다.

## 6개 테스트가 실제로 지나간 경로

| 테스트 | 지나간 주요 분기 |
|---|---|
| 상신후_승인 | `d≠null`→`u≠null`→`proc==1`→`s==0`→`approver≠null`→(재상신)`proc==2`→`s==1`→`approverId 일치`→`role≥2`→`drafter≠null` |
| 상신후_반려 | 위와 동일한 상신 경로 → `proc==3`→`s==1`→`approverId 일치`→`role≥2`→`drafter≠null` |
| 임시저장_취소 | `proc==9`→`s==0`(OR의 첫 항)→`drafterId 일치` |
| 권한없는_승인 | 상신 경로 → `proc==2`→`s==1`→`approverId 일치`→`role≥2`가 **거짓** (유일하게 "거짓" 분기를 검증한 케이스) |
| 없는_id | `d==null` → 즉시 return |
| 재승인 | 상신+승인 경로 반복 → 두 번째 승인 시도에서 `s==1`이 **거짓**(이미 2) |

## 결론

- 6개 테스트는 "정상 경로 + 이 과정에서 요구한 4개 대표 이상 케이스(권한없음/없는id/재승인)"는 정확히 고정했다.
- 다만 **반려·취소의 권한/소유권 검증**은 승인 쪽만큼 대칭적으로 검증되지 않았고(C), **재상신·재반려·상태 역행** 같은
  "같은 액션 재요청" 계열(B)과 **금액 임계값 자동 상향**(D)도 비어 있다.
- 이 표는 처음 요청받은 "6개 고정" 범위를 넘어서는 관찰이므로, 추가 테스트 작성은 이번 문서에서
  실행하지 않았다 — 다음에 필요할 때 이 표의 카테고리(A~E) 단위로 요청하면 골라서 추가할 수 있다.
