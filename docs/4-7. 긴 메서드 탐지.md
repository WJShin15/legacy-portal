# 4-7. 긴 메서드 탐지

전체 소스의 메서드를 대상으로 **줄 수(라인 카운트)** 와 **최대 중첩 깊이**를 측정했다.
(탐지만 수행 — 리팩토링/추출은 별도 단계. `docs/4-4`, `docs/4-5` 참고)

## 측정 기준

- **줄 수**: 메서드 시그니처가 시작하는 줄부터, 대응하는 닫는 중괄호(`}`) 줄까지 포함한 라인 수.
- **중첩 깊이**: 메서드 바디 자체를 깊이 1로 보고, 그 안에 `if`/`for`/`try` 등 블록이 한 겹 중첩될 때마다
  +1. 단, `if / else if / else if ...` 처럼 **같은 층위에서 분기하는 체인**은 깊이를 늘리지 않고
  체인 전체를 한 단계로 계산한다(순수 분기 나열이지 중첩이 아니므로).
- Entity(`Approval`/`Notice`/`Schedule`/`User`)의 getter/setter는 전부 1줄짜리 위임 코드라
  표에서 제외한다(로직이 없어 측정 의미가 없음).

몰려 있다.

## `processApproval` 상세 — 분기별 깊이 분석

86줄짜리 메서드 하나가 상신(1)/승인(2)/반려(3)/취소(9) 네 가지 액션을 전부 처리한다.
분기마다 중첩 깊이가 다르며, 특히 승인·반려 분기가 가장 깊다.

| 분기 (action) | 위치(행) | 조건 중첩 경로 | 최대 깊이 |
|---|---|---|---|
| 상신 (`proc==1`) | 90~109 | `proc==1` → `s==0` → `type==1 && amount≥100만` (또는 `approver!=null`) | 4 |
| **승인 (`proc==2`)** | 110~128 | `proc==2` → `s==1` → `approverId.equals` → `role>=2` → `drafter!=null` | **6** |
| **반려 (`proc==3`)** | 129~148 | `proc==3` → `s==1` → `approverId.equals` → `role>=2` → `drafter!=null` | **6** |
| 취소 (`proc==9`) | 149~159 | `proc==9` → `s==0 \|\| s==1` → `drafterId.equals` | 4 |

승인·반려 두 분기는 조건 경로까지 완전히 동일하다(상태 필드만 다르게 씀) — 이는 `docs/4-5`에서
탐지한 "권한 판정 중복"과 같은 지점이며, 길이·중첩 문제와 중복 문제가 **같은 코드 조각에서
동시에** 발생하고 있음을 보여준다.

## 관찰 — 왜 이 메서드가 유일하게 50줄을 넘는가

- 다른 서비스(`NoticeService.publish`, `ScheduleService.create`)도 중첩은 있지만(깊이 4~5),
  **한 메서드가 하나의 액션만** 처리해서 길이는 50줄 밑에 머문다.
- `processApproval`만 **네 가지 서로 다른 액션(상신/승인/반려/취소)을 하나의 메서드 시그니처
  뒤에 숨겨 action 매직넘버로 분기**하기 때문에 길이와 중첩이 함께 폭증했다.
  (`docs/4-4`에서 확인한 "public 메서드 계약 유지" 불변 규칙상 `processApproval(id, userId, action,
  reason)` 시그니처 자체는 바뀌면 안 되고, 내부에서 분기별로 위임하는 구조가 리팩토링 후보다.)
- `ScheduleService.create`(39줄, 깊이4)는 50줄 문턱에 못 미치지만 "검증 3개 + 겹침판정 루프 +
  저장 + 로그"가 한 메서드에 몰려 있어 **다음으로 긴 메서드**이자 잠재적 후보로 관찰해둔다.
- `NoticeService.publish`(25줄)는 길이는 짧지만 **깊이 5**로, 짧은 메서드에 중첩이 집중된
  사례 — 줄 수만으로는 놓치기 쉬운 케이스라 별도로 표시했다.
