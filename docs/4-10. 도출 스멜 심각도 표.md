# 4-10. 도출 스멜 심각도 표

`docs/4-4`~`4-8`에서 탐지한 코드 스멜을 모아 심각도를 매겼다.

## 이 문서의 성격: AI 산출 목록 = 검증·합의의 기준선

아래 표는 Claude(AI)가 코드를 읽고 판단한 **1차 초안**이다. 심각도 등급과 순위는
"정답"이 아니라 **팀/수업에서 검토하고 토론해 확정해야 할 출발점(기준선, baseline)**
이다. AI가 놓친 맥락(운영 이력, 실제 장애 경험, 비즈니스 우선순위)이 있을 수 있으므로,
아래 등급에 동의하지 않는 항목이 있다면 그 자체가 유의미한 토론 소재다. 이 문서의
목적은 "논쟁의 여지 없는 결론"이 아니라 **"같은 출발선에서 합의를 시작하기 위한 기준선"**
을 제공하는 것이다.

## 심각도 판단 기준

다음 네 가지를 종합해 상/중/하를 매겼다.

1. **영향 범위** — 몇 개의 클래스/메서드에 걸쳐 나타나는가
2. **버그·정합성 위험** — 지금 당장 또는 향후 수정 시 실제 오류로 이어질 가능성
3. **안전망 부재와의 결합** — 테스트가 0개인 상태에서 이 스멜을 건드릴 때의 회귀 위험
4. **후속 리팩토링을 막는 정도** — 이 스멜이 해소되지 않으면 다른 개선이 막히는가

| 등급 | 의미 |
|---|---|
| **상** | 여러 곳에 걸쳐 있거나, 실제 오류로 이어지거나, 다른 리팩토링의 전제조건인 것 |
| **중** | 가독성·유지보수성을 떨어뜨리지만 국소적이고 파급力이 제한적인 것 |
| **하** | 스타일/관례 수준. 지금 당장의 위험은 낮은 것 |

## 스멜 심각도 표

| 순위 | 스멜 | 심각도 | 영향 범위 | 판단 근거 | 관련 문서 |
|---|---|---|---|---|---|
| 1 | **테스트 0개(안전망 부재)** | 상 | 전체 | 이 목록의 다른 모든 항목을 고칠 때 회귀를 잡아줄 안전장치가 없다 — 다른 항목보다 먼저 해소해야 나머지 리팩토링이 안전해진다 | `CLAUDE.md`, `4-4` |
| 2 | **God Class (`ApprovalService`)** | 상 | 클래스 1개, 메서드 7개 | 검증+영속화+메일+로그+포맷팅+권한판정을 한 클래스가 전담. 아래 Long Method·중복·Tight Coupling 항목 다수의 근본 원인 | `4-4`, `4-7` |
| 3 | **Entity 직접 API 노출 + 캡슐화 부재** | 상 | Entity 4종 + Controller 전체 | `Approval`/`Notice`/`Schedule`/`User` 전부 public setter이고, `ApprovalController`가 `Approval` 엔티티를 그대로 응답으로 반환 — 내부 필드가 계약 없이 그대로 노출되고, 외부에서 상태를 서비스 우회로 바꿀 길도 열려 있음 | `4-4` |
| 4 | **Duplicated Code (감사로그·메일본문·권한판정·가드절·상태라벨)** | 상 | 서비스 3개, 6개 패턴 | 같은 규칙(예: `role>=2`)이 여러 곳에 복붙돼 있어 규칙이 바뀌면 일부만 고치고 넘어갈 위험이 실제로 존재 | `4-5` |
| 5 | **Long Method + 중첩 6단계 (`processApproval`)** | 상 | 메서드 1개(86줄) | 상신/승인/반려/취소 4개 액션이 한 메서드에 섞여 있고 최대 깊이 6 — 가독성·리뷰 난도가 가장 높고 회귀 위험도 가장 크다 | `4-7` |
| 6 | **Magic Number / Primitive Obsession 전반** | 상 | 필드 6종 + 파라미터 1종 | `status`/`type`/`priority`/`role`/`action`/금액임계값 등 11개 값 공간이 흩어져 있고, 그중 일부는 서로 다른 개념이 우연히 같은 숫자를 공유(§ action vs status) — 착오 위험이 넓게 퍼져 있음 | `4-8` |
| 7 | **Silent Failure(예외 대신 조용히 무시/`null` 반환)** | 상 (단, 이번 단계 조치 보류) | 서비스 3개 전체 | 권한 없는 처리나 검증 실패를 호출자가 알 방법이 없음 — 설계 결함으로는 심각하지만, `CLAUDE.md` 원칙상 **레거시 동작 보존 대상**이라 이번 리팩토링 단계에서는 고치지 않고 재설계 단계로 이월 | `CLAUDE.md`, `4-2` |
| 8 | **Tight Coupling (협력 객체 직접 `new`)** | 중 | 5곳(서비스 3개) | 테스트 대체 불가로 테스트 작성을 막지만, 그 자체로 런타임 오류를 일으키진 않음 | `4-5` |
| 9 | **Long Parameter List (`create()` 파라미터 8개)** | 중 | 메서드 1개 | 타입이 겹치는 `Long drafterId`/`Long approverId` 순서를 실수로 바꿔도 컴파일러가 못 잡아준다는 점에서 국소적이지만 실제 오류 가능성 있음 | `4-4` |
| 10 | **Poor Naming (`d`, `u`, `proc`, `tmp`, `flag1`, `s`)** | 중 | 서비스 전체 | 동작에는 영향 없지만 `CLAUDE.md`의 최우선 네이밍 규칙과 정면으로 배치되고, 코드 리뷰·온보딩 비용을 크게 높임 | `CLAUDE.md`, 전체 서비스 파일 |
| 11 | **Feature Envy (`amountGrade`, 우선순위 자동상향)** | 중 | 메서드 2개 | `Approval`의 데이터를 꺼내 서비스가 도메인 규칙을 계산 — Rich Domain 전환을 막지만 국소적 | `4-4` |
| 12 | **정의됐지만 로직 없는 사각지대 값** (`type` 2/3/4, `category` 3, `status`=9 미도달) | 중 | 필드 3종 | 지금은 오류를 안 내지만, 향후 그 값을 실제로 쓰는 기능을 추가할 때 "원래 있어야 할 로직이 빠졌다"는 사실을 놓치기 쉽다 | `4-8` |
| 13 | **구조 비대칭 (Notice/Schedule Controller 없음, User Service 없음)** | 하 | 도메인 4개 | 지금은 미노출 상태라 당장의 위험은 낮지만, 향후 이 도메인들을 REST로 열 때 서로 다른 패턴을 베낄 위험 | `4-4` |
| 14 | **Comment Smell (나쁜 이름을 주석으로 변명)** | 하 | 서비스 전체 | Poor Naming(10번)의 증상일 뿐, 독립적인 추가 위험은 크지 않음 | `CLAUDE.md`, `ApprovalService` |
| 15 | **NPE 위험 패턴 (`title.equals("")`)** | 하 | 1곳 | `title`이 `null`이면 그 앞 조건에서 걸러지므로 실제 NPE 발생 경로는 없지만, 패턴 자체는 위험한 관용구 | `ScheduleService.create` |
| 16 | **금액 임계값 100만원 중복** (우선순위 상향 vs 등급 A) | 하 | 메서드 2개 | 우연의 일치일 가능성이 있고 지금은 오작동을 일으키지 않지만, 두 규칙 중 하나만 바뀌면 조용히 어긋날 수 있음 | `4-8` |

## 다음 단계

이 표의 등급·순위를 그대로 실행 계획으로 옮기지 않는다. 팀/수업에서 다음을 확인한 뒤
확정한다.

1. 등급에 이견이 있는 항목 표시 — 특히 "상"으로 매겨진 7개 항목 중 우선순위가 실제
   업무 영향과 맞는지
2. 7번(Silent Failure)처럼 "심각하지만 이번 단계에서는 보존"으로 분류된 항목이
   맞는지 — `CLAUDE.md`의 보존 대상/개선 대상 구분과 재확인
3. 합의된 순서로 실제 리팩토링 단계(테스트 작성 → God Class 분해 → 중복 제거 →
   enum화 → DTO 분리) 진행 — 이 문서는 탐지·합의 기준선이며 실행 계획은 별도 문서
