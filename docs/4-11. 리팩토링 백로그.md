# 4-11. 리팩토링 백로그

`docs/4-4`(구조) · `4-5`(중복코드) · `4-7`(긴 메서드) · `4-8`(매직넘버) · `4-10`(심각도 표)의
진단 결과를 실행 가능한 작업 단위로 옮기고 **우선순위 순서**를 매겼다.
(주의: `docs/smells.md`라는 별도 파일은 존재하지 않아, 위 5개 진단 문서를 통합해 이 백로그의
근거로 삼았다.)

## 순서를 정한 기준 — 심각도 ≠ 실행 순서

`4-10`의 심각도(상/중/하)를 그대로 실행 순서로 쓰지 않았다. 심각도는 "얼마나 나쁜가"를
answer하지만, 백로그 순서는 "무엇을 먼저 해야 다음 작업이 안전한가"(선행조건)를
answer해야 하기 때문이다. 예: Silent Failure(심각도 상)는 심각하지만 `CLAUDE.md`
원칙상 이번 리팩토링 범위에서 **의도적으로 제외**했고, 테스트 부재(심각도 상)는
다른 모든 항목의 **선행조건**이라 항상 1순위다.

## 이번 백로그에서 의도적으로 제외한 항목

| 스멜 | 제외 이유 |
|---|---|
| Silent Failure / 예외 대신 `null` 반환·조용한 무시 (`4-10` #7) | `CLAUDE.md` 원칙상 **레거시 동작 보존 대상**. 지금 고치면 "리팩토링"이 아니라 "동작 변경"이 된다 → 별도 재설계 단계 이슈로 이월 |
| 구조 비대칭 (Notice/Schedule Controller 부재, User Service 부재) (`4-10` #13) | 없는 걸 새로 만드는 건 리팩토링이 아니라 **기능 추가**. 이번 백로그 범위 밖 |
| 사각지대 매직넘버 값에 로직 추가 (`type` 2/3/4 등) | 지금 없는 동작을 새로 넣는 것 = 동작 변경. BL-4에서 enum 값 자체는 정의하되 로직은 추가하지 않음 |
| NPE 위험 패턴(`title.equals("")`), 금액 임계값 100만원 중복 (`4-10` #15, #16) | 심각도 낮고 국소적 — 별도 백로그 항목 대신 BL-3/BL-4 작업 중 **부수적으로 함께 정리** |
